Security Considerations

SQL Injection Prevention

User input must never be inserted directly into SQL strings. Use prepared statements / parameterised queries. This prevents attackers injecting SQL through inputs.

Cross-Site Scripting (XSS) Prevention

Any user content displayed in HTML must be escaped or encoded (especially descriptions). Template engines often escape by default, but it must be verified.

Input Sanitisation

Validate and normalise inputs on the server before storing. Reject invalid or suspicious values. Client-side validation helps usability but is not security.

Server-Side Validation

  • Amount must be a positive number.
  • Type must be one of: income/expense.
  • Category must be a valid category ID.
  • Description should be limited (e.g., max 200 chars).

Example validation logic (pseudocode)

if amount is missing OR amount <= 0:
  return 400 with error message

if type not in ["income", "expense"]:
  return 400 with error message

if categoryId is missing OR not found in DB:
  return 400 with error message

save to DB and redirect (303) to /transactions

Flash Messages

After submitting a form, the system can show temporary messages:

  • Success message when a transaction is added.
  • Error message when validation fails.
  • Message clears after it is shown once (stored in session or cookie).

Responsive Design

  • Use flexible containers and relative units where possible.
  • Use CSS grid/flex layouts that adapt to screen width.
  • Ensure readable font sizes and touch-friendly buttons on mobile.
